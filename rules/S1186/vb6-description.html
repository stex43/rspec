Description

Why is this an issue?

An empty procedure is generally considered bad practice and can lead to confusion, readability, and maintenance issues. Empty procedures bring no functionality and are misleading to others as they might think the procedure implementation fulfills a specific and identified requirement.

Such procedures should be avoided and possibly removed.

Sub DoSomething()
End Sub

However, there are some cases where a procedure needs to be empty. In those cases, it is essential to minimize confusion and enhance clarity.

Here are a few examples:

  • The procedure will never be implemented.

    In order to provide the initial intention to not add an implementation in the future, you can raise an error and specify that the procedure will not be implemented.

    Sub DoSomething()
    Err.Raise(1, , "NotSupported")
End Sub

  • The procedure will be implemented in the future.

    In order to provide the initial intention to add an implementation in the future, you can raise an error and specify that the procedure is not implemented yet.

    Sub DoSomething()
    Err.Raise(1, , "NotImplemented")
End Sub

  • The procedure implementation is empty intentionally.

    In this case, it is important to put a comment explaining the reason for the empty implementation.

    Sub DoSomething()
    ' Do nothing because of X
End Sub
